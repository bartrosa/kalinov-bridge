# ADR-0005 — Lean toolchain for KalinovBridge / LeanProver

## Status

Accepted

## Context

The Kalinov Python stack needs a reproducible Lean 4 environment for
`LeanProver`: stable compiler APIs, Mathlib availability for realistic smoke
tests, and predictable CI behaviour. Toolchains drift quickly; experiments must
be pinned.

## Decision

- Use **[elan](https://github.com/leanprover/elan)** as the single installer for
  `lean` and `lake` (official distribution path for Lean 4).
- Pin the compiler with a committed **`lean-toolchain`** file
  (`leanprover/lean4:v4.30.0-rc2` at the time of this ADR). Bump only with a
  deliberate doc + manifest update.
- Pin **mathlib4** to a **fixed git revision** in `lake-manifest.json`
  (generated by `lake update` from `lakefile.toml`). Reproducibility outweighs
  living at HEAD.
- Prefer **`lake build`** for verification workflows instead of ad hoc `lean`
  invocations without workspace context — Mathlib and transitive deps expect Lake.
  For fast syntax/elaboration smoke checks, **`lake env lean <file>`** is used
  so the file is checked in the workspace environment without a full rebuild when
  appropriate.
- **Timeouts**: subprocess calls use configurable ceilings — defaults **60 s**
  for compile-style checks and **120 s** for full `lake build` checks — so hung
  elaboration cannot stall orchestration indefinitely.

## Consequences

- Developers and CI must install elan (or another coherent toolchain providing
  compatible `lake`/`lean` binaries on `PATH`).
- Bumping Lean or Mathlib is a **small release event**: edit `lean-toolchain`
  and `lakefile.toml`, run `lake update`, commit the manifest, re-run
  `pytest -m lean` and `lake build` under `provers/lean/runtime`.
